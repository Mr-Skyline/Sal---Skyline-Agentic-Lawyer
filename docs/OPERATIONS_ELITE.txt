3) CONTINUOUS SYNC (24/7 style) — disk + optional DB
   WHAT HITS DISK
   - Thread JSON: CORRESPONDENCE_ARCHIVE_DIR\<gmail_thread_id>.json (one file per archived thread).
   - Poll state: correspondence_sync_state.json in project root (last internalDate per thread), written atomically.
   WHAT HITS SUPABASE (only if URL + service key in .env)
   - Table correspondence_threads: metadata upsert per new/updated archive (see db.py / ingest.py).
   SET .env
       AGENT_GMAIL_ADDRESS=the agent's email
       CORRESPONDENCE_ARCHIVE_DIR=folder or UNC path (mapped share drive is fine)
       SYNC_POLL_INTERVAL_SEC=300   (or similar)
       SYNC_NEWER_THAN_DAYS=7       (Gmail query window)
       Optional loop tuning: SYNC_WORKER_BACKOFF_CAP_SEC, SYNC_WORKER_MAX_CONSECUTIVE_ERRORS (see .env.example)
   - Optional local archive bootstrap: py ensure_correspondence_archive.py
   - Preview what would sync (Gmail API; no disk/DB writes; archive dir not required): py sync_worker.py --dry-run
   - Test one cycle: py sync_worker.py --once   OR   sync_worker_once.cmd   (exit 1 if that cycle errors — Task Scheduler)
   - Always on: py sync_worker.py   (backoff between errors; optional max consecutive failures then exit)

   WINDOWS TASK SCHEDULER (polling every N minutes)
   - General tab: choose “Run whether user is logged on or not” if the job must run unattended;
     the account must be able to read credentials.json + token.pickle and write the archive path.
   - Triggers: Daily / repeat task every 5–15 minutes (or your policy), duration “Indefinitely”.
   - Actions → Start a program:
       Program/script:  C:\Windows\System32\cmd.exe
       Add arguments:     /d /s /c "C:\Users\travi\Projects\AI Lawyer Build\sync_worker_once.cmd"
       Start in:          C:\Users\travi\Projects\AI Lawyer Build
     (Adjust the path if your canonical folder differs; keep “Start in” = project root so .env resolves.)

   - sync_worker_once.cmd is non-interactive (no pause). For a long loop instead of scheduled --once,
